# UI-01 桌面端工作台交互问题核对

## 背景

本文基于一次 UI 交互走查结论，并结合当前仓库源码复核后整理。项目桌面端是“多标签工作台 + 左侧功能栏 + 中央项目/词条编辑区 + 右侧 AI/工具停靠面板”的复杂桌面应用，当前主要问题不是缺少页面，而是工作台心智尚未完全收口。

用户需要随时明确：

- 当前所在位置
- 当前编辑对象
- 词条是否已保存
- AI 正在引用哪个项目或词条
- 右侧面板当前是辅助面板还是全屏工作区

## 本轮已落地

- AI 输入超过 4000 字时会截断并提示超出字数。
- 设置页通用配置保存增加“保存中 / 已保存 / 保存失败 / 重试”反馈。
- 右侧停靠面板拖拽时增加“释放后折叠 / 释放后全屏 / 调整宽度”提示。
- 顶部 Tab 不再提供会创建空白“新标签”的加号入口。
- 项目编辑器主内容顶部增加只读 breadcrumb。
- 普通 AI 会话顶栏增加只读上下文条，显示项目、词条、读写模式和联网状态。

## 已被源码证实的问题

### P0-1 词条保存状态不够显性

证据：

- `src/features/entries/components/EntryEditor.tsx` 内部维护 `hasChanges`、`saving`、`autoSaveStatus`，并通过 `onDirtyChange` 通知外层。
- `src/app/desktop/DesktopApp.tsx` 维护 `entryDirtyMap`，关闭词条标签或窗口时会检查未保存状态。
- 当前关闭提示只有“继续关闭会丢失修改”，没有“保存并关闭 / 放弃 / 取消”。
- `flowcloudai-ui` 的 `TabItem` 当前只有 `key`、`label`、`disabled`、`closable` 等基础字段，桌面端传入的 tab label 没有承载 dirty、保存中、保存失败等状态。

建议：

- 在标签标题上显示词条 dirty 点。
- 保存中、保存失败状态应至少在词条页头部固定可见。
- 关闭词条标签或关闭窗口时，提供“保存并关闭 / 放弃更改 / 取消”。
- 自动保存失败时，不应只在编辑器局部文字里弱提示，应进入可见错误状态。

### P0-3 标签导航规则混杂

证据：

- 顶部 Tab 的空白“新标签”入口已取消，不再创建无内容标签。
- 项目、词条、项目工具页都能成为顶层 tab。
- 点击设置会 `setActiveKey('')`、`setMainContentKey('settings')`，并收起右侧面板，容易打断当前工作上下文。

建议：

- 已处理：取消无内容“新标签”。
- `+` 按钮改为明确动作：新建世界观、新建词条、打开命令面板，三选一。
- 设置页应明确是全局页面、独立 tab，还是 modal，不建议清空当前 active tab。
- 项目工具页应决策为“顶层 tab”或“项目内二级页”，不要长期混用。

### P0-4 右侧停靠面板存在隐含拖拽规则

证据：

- `DockableSidePanel.tsx` 中存在 `PANEL_COLLAPSE_THRESHOLD_RATIO = 1 / 5`。
- `DockableSidePanel.tsx` 中存在 `FULLSCREEN_TRIGGER_DISTANCE = 150`。
- 拖拽宽度过小会触发折叠，拖到接近左侧会触发全屏。

修正原走查表述：

- AI 面板顶部已经有显性“全屏 / 退出全屏”和“最小化”按钮。
- 但拖拽阈值触发折叠或全屏仍然比较隐含。

建议：

- 已处理：拖拽时增加状态提示，例如“释放后折叠”“释放后全屏”。
- 统一左侧分类树和右侧面板的拖拽手柄视觉与行为。
- 抽象公共 `SplitHandle` 或 `ResizablePane`。

### P0-5 弹窗层级和焦点管理需要统一

证据：

- 多处 CSS 存在硬编码 `z-index`：500、600、700、1000、1001 等。
- `ProjectCreator` 有 `role="dialog"`、`aria-modal="true"`、`autoFocus`、Esc/backdrop 关闭。
- 但没有统一 Dialog 组件，也没有看到统一 focus trap、return focus 规则。

建议：

- 建立统一 z-index token：popover、dropdown、side panel、modal、global overlay、toast。
- 新增或接入统一 Dialog 组件。
- Dialog 统一支持 focus trap、Esc 关闭规则、关闭后 return focus。
- `ProjectCreator`、封面选择器、词条图片弹窗、类型/标签创建弹窗优先迁移。

## 高频体验问题

### P1-1 AI 当前上下文不够显性

证据：

- `DesktopApp.tsx` 根据 active tab 推导 `aiFocus.projectId` 和 `aiFocus.entryId`。
- `useAiController.ts` 会将项目、词条、读写模式、联网状态写入后端 task context。
- `AIChatContent.tsx` 顶栏已显示插件、当前引用项目、当前引用词条、阅读/编辑模式、联网状态、全屏、最小化等控件。
- 报告会话和角色会话保留特殊展示，普通会话已补基础上下文条。

建议：

- 已处理：AI 顶栏增加上下文条：
  - 项目：xxx
  - 词条：xxx
  - 编辑模式 / 阅读模式
  - 联网开启 / 关闭
- 支持一键解除上下文或锁定当前上下文。
- 角色对话和报告对话保留特殊上下文展示，但基础控件语言保持一致。

### P1-2 设置页自动保存缺少全局反馈

证据：

- `Settings.tsx` 对 `settings` 变化做 500ms 防抖保存。
- 通用设置保存已显示“保存中 / 已保存 / 保存失败 / 重试”状态。
- API Key、提示词模板等局部功能仍保留各自反馈。

建议：

- 已处理：设置页右上角固定显示保存状态。
- 已处理：自动保存失败时保留错误提示，并提供重试。
- 已处理：保存成功后显示“已保存”。

### P1-3 项目编辑器内存在双重导航

证据：

- 顶部全局 tab 管项目、词条、工具。
- 项目内部还有左侧分类树、项目概览、分类词条列表、词条编辑器、工具面板。
- `ProjectEditor.tsx` 内同时处理项目概览、分类选中、工具面板和词条打开。

建议：

- 已处理：项目内部增加 breadcrumb，例如“项目 / 分类 / 词条”或“项目 / 工具 / 时间线”。
- 左树收起后保留项目名或窄 rail，避免用户失去当前位置。

### P1-4 左树和右侧面板拖拽规则重复

证据：

- `ProjectEditor.tsx` 的左树有宽度、折叠阈值和拖拽实现。
- `DockableSidePanel.tsx` 的右侧面板也有宽度、折叠阈值和拖拽实现。

建议：

- 抽象统一分栏拖拽组件。
- 统一 hover、dragging、collapse preview、双击恢复、键盘可达性。

### P1-5 AI 菜单键盘可达性不足

证据：

- AI 插件菜单和模型菜单主要依赖点击和外部点击关闭。
- 当前菜单容器没有 `role="menu"`、触发按钮没有完整 `aria-haspopup` / `aria-expanded`。
- 没有方向键切换、Enter 选择、Esc 关闭、关闭后焦点回退。

建议：

- 插件菜单和模型菜单补齐 ARIA。
- 支持方向键、Enter、Esc。
- 菜单关闭后焦点回到触发按钮。

### P1-6 AI 输入框超限反馈弱

证据：

- `AIChatContent.tsx` 中 `MAX_CHARS = 4000`。
- `SHOW_HINT_THRESHOLD = 3500`。
- 输入超过 4000 时已截断并显示超出字数提示。

建议：

- 已处理：粘贴超过限制时提示“已截断”或“超出 x 字”。
- 已处理：直接输入超限时显示原因。
- 已处理：允许用户明确知道发送按钮不可用或文本被拒绝的原因。

## 视觉系统与组件治理

### P2-1 UI 统一文档已有，但尚未变成强约束

证据：

- `docs/ui_style_unification_plan.md` 已明确要求统一颜色来源、页面骨架、组件状态、尺度系统。
- 文档中明确提出不要在应用层继续新增第二套主题系统和裸色值。

建议：

- 将该文档作为新增 UI 的验收标准。
- PR 或代码评审中检查 token、间距、状态、组件复用。

### P2-2 裸色值和局部视觉体系仍然存在

证据：

- `EntryEditor.css` 中存在 `#6a63ed`、`#8451ed` 等硬编码颜色。
- `AIChatContent.css` 中存在多处 `rgba()`、渐变、局部 z-index。
- 其他功能页中也能检索到裸色和局部渐变。

建议：

- 明确哪些属于品牌例外：封面、角色背景、空状态 hero。
- 常规按钮、导航、表单、状态、卡片、列表统一使用 `--fc-*` 语义 token。
- 缺少 token 时优先补 UI 库，不在业务页面临时造色。

### P2-3 页面骨架已有 CSS 类，但不是正式组件

证据：

- `WorkspaceScaffold.css` 已提供 `.fc-page-shell`、`.fc-page-header`、`.fc-section-card`、`.fc-status-banner`、`.fc-op-panel`、`.fc-op-toolbar`。
- 项目列表、设置页、项目概览、时间线、关系图等页面已经部分使用这些类。

建议：

- 沉淀为 React 组件：
  - `WorkspaceHeader`
  - `SectionCard`
  - `StatusBanner`
  - `FilterBar`
  - `EmptyState`
  - `SplitHandle`
  - `DialogForm`

## 推荐落地顺序

1. 词条保存状态体系：dirty 点、保存中、保存失败、保存并关闭。
2. 标签页规则重构：取消无内容“新标签”，明确设置页和工具页归属。
3. AI 上下文显性化：顶栏显示项目、词条、读写模式、联网状态。
4. 统一分栏拖拽：抽 `SplitHandle`，覆盖左树和右侧面板。
5. 统一弹窗层级和 focus trap：优先覆盖 ProjectCreator、封面选择器、词条图片弹窗。
6. 高频页面视觉治理：ProjectList、ProjectEditor、EntryEditor、AIChatContent、Settings 接入统一组件。
7. 清理裸色和重复 CSS：按 `docs/ui_style_unification_plan.md` 做 token 审查。

## 当前结论

原 UI 走查的大方向成立，大多数问题能在源码中找到证据。当前最关键的产品设计任务是收拢桌面工作台心智：让用户随时知道“我在哪里、正在改什么、保存了吗、AI 正在看什么、右侧面板处于什么模式”。

第一轮不建议重画所有页面，应优先处理保存状态、标签规则和 AI 上下文显性化。

# MOBILE-01 移动端「核心闭环可用」计划（UX 主导版）

> 本文取代早前的工程视角初版。经过一轮 UX 讨论后重写，作为移动端推进的权威计划。

## 背景与定位

- **用户群**：很多设定党是买不起电脑的学生 → 手机要能承载**大部分**功能。
- **关键认知**：「承载大部分功能」≠「长得像桌面」。**能力对齐，不是布局对齐**。桌面是"多标签工作台"，手机是"随身笔记本 + 对讲机"，job 不同，照搬桌面 IA 处处别扭。
- **AI 更靠主轴**：手机上打字填表单反人性、说话自然，而本应用本就 AI-native；对没键盘的学生，**AI 是"键盘替身"**。手机要拿到"完整能力"，靠的是①横向导航代替纵深堆叠 ②AI 帮做手动很痛苦的事。
- **本轮范围**：核心创作闭环可用——项目 → 词条浏览/搜索/编辑 → AI 讨论与**写入** → 灵感速记。重功能（关系图谱/世界地图/时间线/版本管理/插件管理/角色对话）押后（见 Backlog）。
- **性能约束（贯穿全程）**：核心用户用低端安卓机 → 浮层内容按需挂载即时卸载、长列表虚拟化/分页、谨慎用 deck.gl/pixi 等重渲染。

## 设计原则（用户拍板 + 评审补充）

1. **视觉与桌面对齐**，简洁直观的创意工作台。注意：对齐的是**视觉**（token/质感），不是**交互模型**。
2. **顶部按钮 ≤3**。补充：「三个点」只放**低频**操作；中高频（如筛选）用常驻 chip/图标，别让三个点变垃圾桶（否则与原则 3 冲突）。
3. **手势 + 按钮双通道**：手势是加速器，按钮 + 安卓物理返回是**可发现的基线**，不让功能只能靠不可见的手势使出来。
4. **交互嵌套 ≤3 级**：靠「往旁边走」（抽屉/分段/浮层）实现，不靠「往深里堆」。

## 手势语法（全局统一，防肌肉记忆冲突）

- **左屏幕边缘起手的右划 = 返回**（全局唯一含义）。
- **内容区内的横划 = 开/关本页侧边抽屉**。
- 返回永远另有：左上角按钮 + 安卓物理返回键兜底。
- 心智："右划 = 划开我左边的东西"——栈里左边是上一页，项目页里左边是分类树，用**起手位置**区分，不撞。

## 已确认的结构决策

- **词条详情 + 编辑合并为一屏**（查看/编辑切换；"沉浸编辑"是它的编辑态）→ 把主线压进 ≤3 级。
- **主页与世界观列表合并**：项目 Tab 根页即列表 + 顶部"继续上次"strip，不单独占一屏。
- **统一浮层容器**（见下）。
- **AI"浮在内容上"**：用户在构思浮动面板，本轮暂缓；先用「写入确认浮层」作为种子，后续再演进为完整浮动面板。

## 浮层容器（基础设施）

- **一个共享 Overlay 基座 + 两个薄变体**：`BottomSheet`（上滑、拖拽下滑关闭、可 snap 档位）/ `FloatingPanel`（四边不挨屏、居中留边、点背板关闭）。
- **对外单一调用入口** `present({ kind, content, ... })`：该统一的统一（portal/背板/动画/焦点/安全区/视觉/调用），差异化（手势/布局）用组合，不做 `variant` 大组件内部到处 if。
- **挂载策略——"舞台常驻、布景按需"**：容器常驻（关闭时是空 portal 宿主，几乎零开销）；**内容打开才挂、关了就卸**；仅对需保状态的（如流式输出中的 AI 对话）保活。移动端默认比桌面更激进地卸载（低端机内存）。
- **先建在 `app_main`**（快、无 rebuild），稳定后再考虑提进 `lib_ui`（注意 app_main 以 `file:../lib_ui/ui` 依赖，改库需 rebuild）。证据：`lib_ui/ui/src/index.ts` 目前**无任何移动原语**（无 Drawer/Sheet/通用 Modal）。
- **拿 2-3 个真实用例喂它，别空中抽象**：① `AiConfirmModal`（浮动）② 一个底部 sheet（词条快捷操作 或 AI 的"+"菜单）。
- **先定栈规则**：同一时刻单活跃浮层，或明确的浮层栈 + 统一 z-index token，避免层级混乱。
- 现状痛点（证据）：`features/ai-chat/components/AiConfirmModal.tsx` 自带 `.acm-overlay`、`ProjectCreator`/导入冲突弹窗各写一套，浮层到处手搓。

## 分期路线（walking skeleton 打法）

打法：不先憋全套设计稿、也不一上来大改；**立地基 + 用一条真实路径串通验证**。对非专业设计师，摸到能跑的薄切片比看静态稿更能判断方向。

**第一阶段要打通的那条线**：
进项目 → 打开词条（合并后的一屏）→ 叫 AI → AI 提议写入 → 新浮层确认 → 落库。
（一条线同时验证：详情/编辑合一 + 浮层容器 + 修掉"AI 在手机上写不了数据"的 P0。）

## 界面草案（用户设计，逐屏）

- **首屏/主页**：继续工作、快速开始、"我的世界观"列表入口。
- **世界观列表**：顶部左返回 / 右"三个点"+新建；下方搜索栏；横向卡片；下拉刷新；右划返回；卡片**左划**弹出 置顶 / 导出 / 删除。
- **世界观页（= ProjectEditor）**：左上返回 + 分类树开关；右上三个点（预留）；Overview 精简；分类 + Tags 收进悬浮卡片只留入口（避免页面过高）；下方全部词条**两列**；页面**右划**→分类树抽屉（主内容右移并适量缩小）；CategoryView 逻辑同桌面，筛选放右上三个点。
- **词条编辑**：考虑竖屏，正文有沉浸编辑模式。
- **AI 对话**：进入直接是对话界面；输入区从左到右 = `+` / 模式选择 /（间隔）用量统计 / 发送；左上三横线、或页面右划 → 聊天列表（顶部搜索 + 类型筛选 + 列表，动画与分类树统一）；右上三个点 + 新建对话；三个点或左划 → 当前对话属性设置；模型/插件切换、上传文件收进 `+`。

### 评审备注（待抠细节，不阻塞）

- **用量统计移出输入行**——它是信息不是动作，别占拇指黄金区；挪到对话设置或顶部小标。
- **三个点别变垃圾桶**——只放低频，筛选等中高频用常驻 chip。
- **右划"返回 vs 抽屉"** 用起手位置区分（边缘=返回 / 内容=抽屉），否则冲突。
- **两列词条列表** 遇长中文标题/摘要的换行与可扫描性需验证。
- **"导出"作为卡片左划项是否过重**——左划宜放高频（置顶/删除），导出可移到卡片三个点或项目页。

## 当前结论

移动端不缺页面，缺的是"最后一公里 + 正确的移动心智"。**Step 0 + 阶段 1（浮层地基 + AI 写入 P0）已实现；阶段 2 进行中**（M2-2 词条合一屏 + M2-4 标题收口已落地，剩 M2-1 继续上次条 / M2-3 手势）。重功能一律押后，避免战线拉长。早前工程版里"单独挂 AiConfirmModal"的做法，已并入"浮层容器的首个真实用例"。
